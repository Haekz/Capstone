==========================================================================
                      CONTEXTO DEL PROYECTO: LBWUS
==========================================================================

Este archivo sirve para guiar a cualquier Inteligencia Artificial o desarrollador
que trabaje en este proyecto sobre su arquitectura, sus reglas de diseño y su
filosofía de trabajo.

--------------------------------------------------------------------------
1. DESCRIPCIÓN GENERAL DEL PROYECTO
--------------------------------------------------------------------------
* Nombre: LBWUS (Learn Better With Us)
* Tipo: Plataforma web universitaria de auto-aprendizaje colaborativo para alumnos,
  profesores y tutores.
* Framework: Django (Python).
* Base de datos: SQLite para desarrollo local (db.sqlite3) y PostgreSQL en producción.

--------------------------------------------------------------------------
2. FILOSOFÍA DE DESARROLLO (LIMITACIONES Y REGLAS DE DISEÑO)
--------------------------------------------------------------------------
* SOLO HTML Y CSS (VANILLA CSS):
  No se utilizan frameworks CSS complejos de configurar (como TailwindCSS). Se prefiere
  escribir archivos .css tradicionales y cargarlos mediante {% static %}.
  
* CLASES SENCILLAS Y LEGIBLES:
  Los nombres de las clases en el HTML deben ser planos, descriptivos y sumamente simples
  para que el usuario pueda buscarlos y editarlos fácilmente en las hojas de estilo.
  Evitar anidaciones excesivas o selectores complejos.
  Ejemplos correctos: `.prof-card`, `.prof-input`, `.prof-btn`, `.panel-card`, `.panel-table`.

* FÁCILMENTE MODIFICABLE:
  El código de estilos debe estar estructurado para que una persona con conocimientos básicos
  de HTML y CSS pueda modificar márgenes, colores, tipografías y tamaños de fuente sin esfuerzo.

* SISTEMA DE COLORES Y MODO OSCURO:
  El diseño utiliza variables CSS declaradas en `:root` (dentro de alumnos/static/alumnos/css/styles.css).
  Variables principales:
    * `--primary`: #a855f7 (Morado corporativo)
    * `--primary-dark`: #9333ea
    * `--accent`: #f59e0b (Amarillo/Naranja para CTAs)
    * `--text`: #111827 (Texto oscuro)
    * `--white`: #ffffff (Fondos claros)
  El modo oscuro se maneja agregando la clase `body.dark-mode`. Cualquier hoja de estilo nueva
  debe implementar selectores `body.dark-mode .clase` para asegurar que el texto siga siendo
  legible y los colores coherentes al alternar el modo día/noche.

--------------------------------------------------------------------------
3. ESTRUCTURA DE APLICACIONES (APPS)
--------------------------------------------------------------------------
* alumnos:
  Páginas públicas (Home, Nosotros, Planes, Servicios, Contactos, Simulador, Selección de tipo de usuario) e inicio de sesión y registro de alumnos.
* admin_portal:
  Panel de administración central (`dashboard_admin`), CRUD de alumnos (`crud`, `alumnos_Add`, `alumnos_del`, `alumnos_findEdit`, `alumnos_Update`), gestión de contenidos de páginas administrativas y resolución de reportes (`resolver_reporte`). Protegido por sesión de administrador (`admin_id`).
* user_profesor:
  Registro, inicio de sesión passwordless y panel de profesores (`registro_profesor`, `login_profesor`, `panel_profesor`, `actualizar_perfil`).
  Registro e inicio de sesión passwordless para administradores/tutores usando el modelo `Tutor` (`registro_tutor`, `login_admin`), el cual guarda `admin_id` en la sesión para dar acceso a `admin_portal`.

--------------------------------------------------------------------------
4. MODELO DE DATOS Y FLUJOS ESPECIALES
--------------------------------------------------------------------------
* Modelos Clave (alumnos/models.py):
  `Genero`, `Tutor`, `Alumno`, `Profesor`, `Clase`, `Inscripcion`, `Reporte`.
  
* Flujo de Acceso para Profesores (Passwordless):
  El profesor no requiere contraseña convencional. Para iniciar sesión:
  * Registro Obligatorio: Rellena el formulario con sus datos.
  * Acceso Rápido: Ingresa su RUT o Correo. Si existe, se guarda su ID en la sesión (`request.session['profesor_id']`) y accede al panel del profesor.

* Flujo de Acceso para Administradores (Passwordless):
  El administrador (modelo `Tutor`) se registra y accede con su RUT o Correo Electrónico. Si existe, se guarda el ID de Tutor en la sesión (`request.session['admin_id']`) y accede a `/admin_portal/dashboard/`.

* Gestión de Reportes:
  Los usuarios pueden reportar incidencias mediante el modelo `Reporte`. El administrador puede verlos en su dashboard y cambiarlos a estado 'resuelto' en `/admin_portal/resolver_reporte/<id>/`.

--------------------------------------------------------------------------
5. DESPLIEGUE Y ENTORNO
--------------------------------------------------------------------------
* Entorno Local: Utiliza SQLite (`db.sqlite3`) para el desarrollo local rápido.
* Entorno de Producción: Desplegado en Railway (`capstone-production-eddd.up.railway.app`). Configurado para usar PostgreSQL dinámicamente mediante la variable `DATABASE_URL` y WhiteNoise para servir archivos estáticos de forma eficiente.
